# ISO/IEC 27001 -- Information Security Management

## What Is ISO 27001 and Why It Matters

ISO/IEC 27001 is the international standard for information security management systems (ISMS). It provides a systematic approach to managing sensitive company information, ensuring it remains secure through people, processes, and technology.

**Why it matters for banking GenAI**:
- Many global banks are ISO 27001 certified and require vendors to comply
- It provides a comprehensive security framework beyond any single regulation
- It is recognized globally (unlike region-specific frameworks)
- The 2022 revision (ISO/IEC 27001:2022) includes updated controls relevant to cloud, AI, and data privacy

**Certification process**:
- Stage 1 audit (document review)
- Stage 2 audit (implementation review)
- Surveillance audits annually
- Recertification every 3 years

## ISO 27001:2022 Structure

The standard follows the Annex SL high-level structure:
1. Scope
2. Normative references
3. Terms and definitions
4. Context of the organization
5. Leadership
6. Planning
7. Support
8. Operation
9. Performance evaluation
10. Improvement

Clauses 4-10 are mandatory for compliance. Annex A contains the reference controls.

## Key Controls in Detail -- Engineering Implementation

### A.8.4: Access to Source Code

**Requirement**: Access to source code, development tools, and testing environments must be appropriately restricted.

**Engineering implementation**:
```
REQUIRED CONTROLS:
├── Repository access based on least privilege
├── Branch protection rules (no direct push to main)
├── Required code reviews (minimum 2 approvers)
├── CODEOWNERS file for critical paths
├── Secret scanning enabled on all repositories
├── Signed commits required
├── No hardcoded credentials in source code
└── Regular access review for repository access
```

### A.8.11: Data Masking

**Requirement**: Data masking must be used in accordance with the organization's data classification scheme and other topic-related access control policies.

**Engineering implementation for AI systems**:
```python
class DataMasking:
    """Mask sensitive data in non-production environments."""

    MASKING_RULES = {
        "customer_name": {"method": "pseudonymize", "algorithm": "FF3-1"},
        "email": {"method": "hash", "algorithm": "SHA-256 with salt"},
        "phone": {"method": "mask", "pattern": "***-***-XXXX"},
        "pan": {"method": "mask", "pattern": "****-****-****-XXXX"},
        "ssn": {"method": "mask", "pattern": "***-**-XXXX"},
        "address": {"method": "generalize", "level": "city"},
        "transaction_amount": {"method": "perturb", "epsilon": 0.1},
        "date_of_birth": {"method": "generalize", "level": "year"},
    }

    def mask_dataset(self, dataset: DataFrame, environment: str) -> DataFrame:
        """Apply masking rules based on environment."""
        if environment == "production":
            return dataset  # No masking in production

        masked = dataset.copy()
        for column, rules in self.MASKING_RULES.items():
            if column in masked.columns:
                masked[column] = self.apply_mask(masked[column], rules)
        return masked
```

### A.8.15: Logging

**Requirement**: Logs that record activities, exceptions, and faults shall be produced, stored, protected, and analyzed.

**Engineering implementation**:
```
LOGGING REQUIREMENTS:
├── All authentication events (success and failure)
├── All access to sensitive data
├── All administrative actions
├── All security configuration changes
├── All system errors and exceptions
├── All data processing activities
├── Log format: structured JSON with timestamps
├── Log protection: append-only, integrity verified
├── Log retention: minimum 1 year, 3 months immediately available
├── Log review: automated analysis with alerting
└── Log synchronization: NTP across all systems
```

### A.8.24: Use of Cryptography

**Requirement**: Rules for the use of cryptography, including cryptographic key management, shall be defined and implemented.

**Engineering implementation**:
```
CRYPTOGRAPHIC STANDARDS:
├── Data at rest: AES-256-GCM
├── Data in transit: TLS 1.2+ (TLS 1.3 preferred)
├── Password hashing: Argon2id or bcrypt (cost factor >= 12)
├── Digital signatures: RSA-3072+ or ECDSA P-256+
├── Key management: HSM or cloud KMS
├── Key rotation: annually or upon compromise
├── Certificate management: automated renewal (ACME)
└── Cryptographic inventory: maintained and reviewed annually
```

### A.8.28: Secure Coding

**Requirement**: Secure coding principles shall be applied during software development.

**Engineering requirements for AI systems**:
```
SECURE CODING PRINCIPLES:
├── Input validation (never trust user input)
├── Output encoding (prevent XSS, injection)
├── Parameterized queries (prevent SQL injection)
├── Principle of least privilege
├── Defense in depth
├── Fail securely (default deny)
├── Keep it simple (reduce attack surface)
├── Don't reinvent the wheel (use vetted libraries)
├── AI-specific:
│   ├── Validate AI inputs (prompt injection prevention)
│   ├── Sanitize AI outputs (before displaying or executing)
│   ├── Rate limit AI endpoints
│   ├── Implement content filtering
│   └── Log AI decisions for auditability
```

## Statement of Applicability (SoA)

The SoA is a mandatory document that:
- Lists all 93 Annex A controls
- Indicates whether each control is implemented or not
- Provides justification for exclusions

## Risk Assessment and Treatment

ISO 27001 requires a systematic risk assessment approach:

### Risk Assessment Process

1. **Establish risk assessment criteria**
   - Define risk acceptance criteria
   - Define risk scoring methodology
   - Define risk treatment options

2. **Identify risks**
   - Identify assets, threats, vulnerabilities
   - Identify existing controls
   - Identify consequences of compromise

3. **Analyze risks**
   - Assess likelihood and impact
   - Calculate risk level
   - Compare against risk acceptance criteria

4. **Evaluate risks**
   - Prioritize risks
   - Determine which risks need treatment

5. **Treat risks**
   - Select treatment option (mitigate, transfer, accept, avoid)
   - Select controls from Annex A or elsewhere
   - Document risk treatment plan

## Internal Audit Program

ISO 27001 requires regular internal audits to verify the ISMS is functioning correctly:

```
INTERNAL AUDIT SCHEDULE:
├── Annual audit plan covering all ISMS areas
├── Audit frequency based on risk (high-risk areas audited more frequently)
├── Independent auditors (not auditing own work)
├── Audit findings documented with corrective actions
├── Corrective action tracking and follow-up
└── Management review of audit results

AUDIT AREAS FOR GENAI SYSTEMS:
├── Access controls (quarterly)
├── Change management (quarterly)
├── Incident management (quarterly)
├── Data handling (quarterly)
├── Vendor management (semi-annual)
├── Risk assessment (annual)
├── Business continuity (annual)
└── Policy compliance (annual)
```

## Common Interview Questions

### Question 1: "What is the Statement of Applicability and why does it matter?"

**Good answer**:
The SoA is a mandatory ISO 27001 document that lists all 93 Annex A controls, indicates which are implemented and why, justifies any exclusions, and references related policies. It matters because it's the auditor's roadmap -- they use it to plan their audit. An incomplete or inaccurate SoA suggests the organization doesn't understand its own security posture.

### Question 2: "How do you implement A.8.11 (Data Masking) for an ML training dataset?"

**Good answer structure**:
1. Classify all data fields by sensitivity
2. Define masking rules for each sensitivity level (pseudonymize, hash, mask, generalize, perturb)
3. Apply masking in the data pipeline before data reaches non-production environments
4. Use format-preserving encryption for fields that need to maintain their format for ML
5. Maintain a mapping table in a secure vault for reversible masking
6. Validate masking effectiveness before data is used for training in non-prod
7. Document the masking approach in the data processing records

### Question 3: "What's new in ISO 27001:2022 compared to 2013?"

**Good answer**:
The 2022 revision reorganized controls from 14 domains to 4 themes (Organizational, People, Physical, Technological). It reduced controls from 114 to 93 (through consolidation). New controls include: threat intelligence, information security for cloud services, ICT readiness for business continuity, physical security monitoring, configuration management, information deletion, data masking, data leakage prevention, monitoring activities, web filtering, and secure coding. The emphasis on cloud security and data protection is significantly increased.
