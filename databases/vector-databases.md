# Vector Database Landscape and Selection

## Overview

Vector databases store and search high-dimensional embeddings generated by ML models. For banking GenAI platforms, vector databases power semantic search, document retrieval for RAG, and similarity-based fraud detection. This guide compares vector database options and provides selection criteria for banking environments.

## Vector Database Landscape

| Database | Type | Scale | Index Types | Best For | Banking Fit |
|----------|------|-------|-------------|----------|-------------|
| pgvector | Postgres Extension | ~10M vectors | HNSW, IVFFlat | Co-located with metadata | Excellent |
| Milvus | Dedicated | Billions | IVF, HNSW, SCANN, DiskANN | Large-scale standalone | Good |
| Pinecone | Managed SaaS | Billions | Proprietary | Zero-ops teams | Poor (data residency) |
| Weaviate | Dedicated/Managed | Hundreds of millions | HNSW | Hybrid search | Moderate |
| Qdrant | Dedicated | Hundreds of millions | HNSW | Filtered search | Good |
| Chroma | Embedded/Server | Millions | HNSW, Flat | Development/prototyping | Poor |
| Elasticsearch | Search Engine | Hundreds of millions | HNSW, Int8 | Hybrid text + vector | Moderate |
| Redis (RediSearch) | In-Memory | Millions | HNSW, Flat | Low-latency cache+vector | Good for small scale |

## Detailed Comparison

### pgvector (Recommended for Banking)

```sql
-- Pros:
-- 1. Single database for vectors + metadata + relational data
-- 2. ACID transactions with vector operations
-- 3. SQL joins between vectors and relational data
-- 4. Familiar operational patterns (backup, monitoring, HA)
-- 5. No additional infrastructure to manage
-- 6. Data stays within banking network

-- Cons:
-- 1. Limited scale (~10M vectors per instance)
-- 2. HNSW build time for large datasets
-- 3. PostgreSQL WAL growth with vector updates

-- Best for:
-- RAG with < 10M document chunks
-- Teams already using PostgreSQL
-- Environments with strict data residency requirements

CREATE TABLE document_chunks (
    chunk_id UUID PRIMARY KEY,
    content TEXT,
    embedding vector(1536),
    metadata JSONB
);

CREATE INDEX ON document_chunks USING hnsw (embedding vector_cosine_ops);

-- Hybrid: Vector search + metadata filter
SELECT content, metadata, 1 - (embedding <=> $1) AS similarity
FROM document_chunks
WHERE metadata @> '{"doc_type": "product_info"}'
ORDER BY embedding <=> $1
LIMIT 5;
```

### Milvus

```python
# Pros:
# 1. Massive scale (billions of vectors)
# 2. Distributed architecture with horizontal scaling
# 3. Multiple index types for different trade-offs
# 4. GPU acceleration support
# 5. Mature ecosystem with SDKs

# Cons:
# 1. Additional infrastructure to manage
# 2. Metadata queries less flexible than SQL
# 3. Operational complexity (etcd, MinIO, Milvus)
# 4. Separate backup and DR procedures

# Best for:
# Vector counts exceeding 10M+
# Dedicated ML infrastructure team
# High-throughput, low-latency requirements

from pymilvus import connections, Collection, FieldSchema, CollectionSchema, DataType

connections.connect("default", host="milvus-host", port="19530")

fields = [
    FieldSchema(name="chunk_id", dtype=DataType.VARCHAR, is_primary=True, max_length=100),
    FieldSchema(name="embedding", dtype=DataType.FLOAT_VECTOR, dim=1536),
    FieldSchema(name="doc_type", dtype=DataType.VARCHAR, max_length=50),
    FieldSchema(name="content", dtype=DataType.VARCHAR, max_length=4000),
]

schema = CollectionSchema(fields, "Document chunks for RAG")
collection = Collection("document_chunks", schema)

# Create HNSW index
index_params = {
    "index_type": "HNSW",
    "metric_type": "COSINE",
    "params": {"M": 16, "efConstruction": 256},
}
collection.create_index("embedding", index_params)

# Search
collection.load()
results = collection.search(
    data=[query_embedding],
    anns_field="embedding",
    param={"metric_type": "COSINE", "params": {"ef": 100}},
    limit=5,
    expr="doc_type == 'product_info'",
)
```

### Elasticsearch (Hybrid Search)

```json
// Pros:
// 1. Best-in-class full-text search
// 2. Hybrid text + vector search
// 3. Rich filtering and aggregation
// 4. Mature ecosystem and tooling

// Cons:
// 1. Vector search is secondary feature
// 2. Memory-intensive
// 3. Complex operational overhead

// Best for:
// When full-text search is the primary requirement
// Already running Elasticsearch for logging/search

PUT /document_chunks
{
  "mappings": {
    "properties": {
      "content": { "type": "text", "analyzer": "english" },
      "embedding": {
        "type": "dense_vector",
        "dims": 1536,
        "index": true,
        "similarity": "cosine"
      },
      "metadata": { "type": "object" }
    }
  }
}

// Hybrid search: Text + Vector
POST /document_chunks/_search
{
  "knn": {
    "field": "embedding",
    "query_vector": [0.1, 0.2, ...],
    "k": 10,
    "num_candidates": 100,
    "filter": { "term": { "metadata.doc_type": "product_info" } }
  }
}
```

## Selection Framework

```python
"""Vector database selection decision tree."""

def choose_vector_database(requirements: dict) -> str:
    """Choose vector database based on requirements."""
    
    vector_count = requirements.get('vector_count', 0)
    needs_sql_joins = requirements.get('needs_sql_joins', False)
    data_residency = requirements.get('data_residency', 'on-premise')
    ops_capacity = requirements.get('ops_capacity', 'medium')  # low, medium, high
    needs_hybrid_text_search = requirements.get('needs_hybrid_text_search', False)
    existing_elasticsearch = requirements.get('existing_elasticsearch', False)
    latency_require_ms = requirements.get('latency_require_ms', 100)
    
    # SaaS ruled out for strict data residency
    if data_residency == 'on-premise':
        # No Pinecone, no managed services
    
    # SQL joins needed -> pgvector
    if needs_sql_joins and vector_count < 10_000_000:
        return 'pgvector'
    
    # Hybrid text search needed
    if needs_hybrid_text_search:
        if existing_elasticsearch:
            return 'elasticsearch'
        elif vector_count < 5_000_000:
            return 'pgvector'  # With full-text search extension
        else:
            return 'weaviate'
    
    # Very large scale
    if vector_count > 10_000_000:
        if ops_capacity == 'high':
            return 'milvus'
        else:
            return 'qdrant'  # Easier to operate than Milvus
    
    # Moderate scale
    if vector_count > 1_000_000:
        return 'pgvector'  # With proper hardware
    
    # Small scale
    return 'pgvector'  # Default for banking

# Banking recommendation:
# Most banking GenAI workloads: pgvector
# Exceptions: Very large scale (> 10M vectors) -> Milvus
#             Already have Elasticsearch -> Use it
```
